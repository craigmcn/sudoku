# Sudoku

A browser-based Sudoku game built with TypeScript and Vite, deployed to Netlify. No runtime dependencies — pure vanilla TS.

## Architecture

The project is split into TypeScript modules plus static HTML/CSS:

- **[src/solver.ts](src/solver.ts)** — Core types and algorithms. Exports `Board` (9×9 grid of `number | null`), `isValid`, `solve` (backtracking solver), and `countSolutions` (stops early at `max`, restores the board on return).
- **[src/rng.ts](src/rng.ts)** — `mulberry32(seed)`, a small seedable PRNG returning a `Math.random`-compatible `() => number` in `[0, 1)`. Exists so puzzle generation can be made reproducible from a seed (daily puzzles, puzzle-ID hashing) without pulling in a larger dependency.
- **[src/puzzleId.ts](src/puzzleId.ts)** — `hashSolution(board)`, a stable content-hash ID for a solved grid (two 32-bit FNV-1a passes, concatenated as base36). Used to key puzzle documents so two independently-generated puzzles with the same solution collapse onto the same ID/stats instead of duplicating.
- **[src/generator.ts](src/generator.ts)** — Generates puzzles. Fills a board with a randomized backtracking fill, then removes cells one-by-one while verifying unique solution via `countSolutions`. Clue counts per difficulty: easy 46, normal 36, hard 28, expert 22. `generatePuzzle(difficulty, seed?)` — an optional numeric `seed` makes the whole generation (fill order and removal order) deterministic via `mulberry32`; omitting it falls back to `Math.random` as before.
- **[src/game.ts](src/game.ts)** — Pure functional game state. `GameState` holds the puzzle, solution, `puzzleId` (content hash of the solution, from `hashSolution`), user board, given-cell mask, notes (bitmask per cell), notes mode, timer fields, mistake count, and undo history (`Snapshot[]`). All state mutations return a new `GameState`. Entering a correct number auto-clears that digit from peers' notes. `createGame(difficulty, seed?)` — an optional `seed` reproduces a specific puzzle (daily puzzles); omitting it generates a fresh random one as before.
- **[src/firebase.ts](src/firebase.ts)** — Initializes the Firebase app/auth/db from `VITE_FIREBASE_*` env vars, guarded against re-`initializeApp()`. `auth`/`db` are `Auth | undefined` / `Firestore | undefined` — initialization is wrapped in `try`/`catch` so a missing/invalid config (no env vars set, e.g. CI or a fresh checkout without `.env.local`) degrades to "stats disabled" instead of throwing and crashing every module that imports it, `main.ts` included.
- **[src/puzzleDoc.ts](src/puzzleDoc.ts)** — `boardToRows(board)` and `requireDb()`, plus `ensurePuzzleDoc(puzzleId, difficulty, puzzle, solutionHash, options?)` (creates `puzzles/{puzzleId}` on first sight, no-ops if it already exists; `options.isDaily`/`options.dailyDate` tag daily-puzzle docs). Extracted from `stats.ts` so `recordPuzzleStart` and the daily-puzzle cache (`dailyPuzzle.ts`) share one doc-creation path instead of two copies.
- **[src/stats.ts](src/stats.ts)** — `ensureAnonymousAuth()` (idempotent `signInAnonymously`, memoized as a module-level promise; rejects immediately if `auth` is undefined; does not cache a rejection, so a later call retries), `resetAnonymousAuthCache()` (clears that memoized promise — called by `signOutUser()` in `auth.ts` so the next call re-authenticates instead of reusing a now-stale user), `recordPuzzleStart(puzzleId, difficulty, puzzle, solutionHash)` (calls `ensurePuzzleDoc`, then increments `timesPlayed`), `recordPuzzleCompletion(puzzleId, elapsedMs)` (increments `completions`/`totalPlayTimeMs`), `recordUserPlay(puzzleId, difficulty, mistakes, elapsedMs)` (`setDoc`s `users/{uid}/plays/{puzzleId}` — overwrites, doesn't merge, so a repeat play of the same puzzle replaces its history entry rather than accumulating one). All throw via `requireDb()` if `db` is undefined. Failures are caught and logged in `main.ts`, never block gameplay.
- **[src/dailyPuzzle.ts](src/dailyPuzzle.ts)** — `seedFromString(str)` (single-pass FNV-1a → uint32), `todayUtc(now?)` (UTC `yyyy-mm-dd`), `dailySeed(date, difficulty)`, `dailyRandomDifficulty(date)` (deterministically picks from `DAILY_RANDOM_POOL` = easy/normal/hard, never expert), `generateDailyPuzzle(date, difficulty)` (calls `generatePuzzle(difficulty, dailySeed(...))` — fully local/deterministic, no Firestore read needed to play), `cacheDailyPuzzles(date)` (best-effort: `ensurePuzzleDoc`s all four difficulties tagged `isDaily`, then creates `dailyPuzzles/{date}` pointing at their ids + `random`; tolerates a `permission-denied` race from another client's concurrent create, since the value would be identical by construction), `MIN_CALENDAR_DATE` (the earliest date the calendar view, #41, can browse to — the date daily-puzzle support itself shipped), and `dailyPuzzleId(date, difficulty)` (memoized `hashSolution(generateDailyPuzzle(...).solution)`, used by the calendar's completion scan — memoization matters since `'expert'` generation costs ~130ms per call).
- **[src/calendarView.ts](src/calendarView.ts)** — Pure, DOM/Firestore-free helpers for the calendar overlay (#41), mirroring `statsView.ts`'s separation of pure logic from `main.ts`'s DOM wiring: `buildCalendarMonth(year, month, today, minDate, completedDates)` (always a 42-cell Sunday-start grid; a day's `status` is `'completed' | 'not-completed'` when selectable, `'unavailable'` otherwise), `isDateSelectable`/`selectableDatesInMonth` (lexicographic `yyyy-mm-dd` range checks against `[minDate, today]`), `canGoToPreviousMonth`/`canGoToNextMonth`/`shiftMonth` (month-nav bounds), `formatMonthLabel`, and `difficultiesWithCompletions(plays)` (the set of difficulties a user has ever completed at all, used to skip expensive puzzle generation for difficulties with zero completions).
- **[src/auth.ts](src/auth.ts)** — Real login on top of #17's anonymous identity. `onAuthChange(callback)` (live `onAuthStateChanged` subscription for UI; no-ops instead of throwing if `auth` is undefined, since it runs unconditionally at boot rather than from a user action). `signInWithGoogle()` and `completeEmailLinkSignInIfPresent()` (the redirect-back half of `sendSignInLink(email)`) both **link** rather than replace when the current session is anonymous — `linkWithPopup`/`linkWithCredential` upgrade the existing anonymous uid in place, so play stats already written under it (via `stats.ts`) carry over into the real account instead of being orphaned. Both fall back to `signInWithCredential` on `auth/credential-already-in-use` (the same Google account/email is already tied to a different Firebase user, e.g. signed in on another device) — that fallback signs into the existing account rather than merging, abandoning the current anonymous session's stats; a deliberate, documented tradeoff, not a bug. `signOutUser()` calls `signOut` then `resetAnonymousAuthCache()`.
- **[src/persistedGame.ts](src/persistedGame.ts)** — `saveGame(state)`, `loadGame()`, `clearSavedGame()`. Serializes the in-progress `GameState` to a single `localStorage` key (`sudoku-saved-game`) so a backgrounded tab that the browser discards and reloads from scratch can resume instead of silently starting over. `difficulty` is read from `state.difficulty` rather than taken as a separate parameter (see key decisions below). Versioned (`SCHEMA_VERSION`) and age-limited (24h) — a save that fails either check, or is unparseable, is discarded rather than rehydrated. Never persists (and clears any prior save for) a solved game; strips `history` (the undo stack) before saving. All `localStorage` access is wrapped in `try`/`catch`, matching this repo's "never let storage/backend issues block gameplay" pattern from `firebase.ts`/`stats.ts`.
- **[src/statsView.ts](src/statsView.ts)** — Pure, DOM-free helpers for the stats/profile view (#27): `summarizeByDifficulty(plays)` (per-difficulty completion count, best time, average time — `null` best/avg when a difficulty has zero completions), `formatElapsed(ms)` (mm:ss), `formatDifficultyLabel(difficulty)` (capitalized). Kept separate from `src/stats.ts` (which does the Firestore I/O) the same way `game.ts`/`dailyPuzzle.ts` separate pure logic from `main.ts`'s DOM wiring — so the aggregation math is unit-testable without mocking Firestore.
- **[src/main.ts](src/main.ts)** — DOM wiring. Builds the board and numpad on init, handles clicks and keyboard input (digits, Backspace/Delete/0 to erase, `N` for notes mode, arrow keys to navigate). Renders all cell classes (given, selected, peer, same-num, conflict, has-value, has-notes) and dims completed digits in the numpad. Shows a loading overlay during puzzle generation (yields to the browser with a 30ms timeout before the sync generation work). Calls `recordPuzzleStart` when a new game is created and `recordPuzzleCompletion`/`recordUserPlay` (once, guarded by a module-level flag) when the board is solved. `btnDaily`/`btnDailyRandom` start today's puzzle via a module-level `activeSeed` passed through to `createGame`; any regular difficulty pick, New Game, or Play Again clears it back to random generation. `render()` calls `saveGame` on every state change; `visibilitychange`/`pagehide` listeners save explicitly too, since the timer's per-second tick updates `state.elapsed` without going through `render()`. `init()` tries `tryRestoreGame()` before falling back to `startNewGame()` — a restored game that had already started comes back `paused` (with the pause overlay shown) regardless of its saved pause state, so resuming after a surprise reload is always an explicit action, not an instantly-ticking timer. Sign-in UI: `#btnSignIn` opens `#signInOverlay` (Google popup or email-link form); `onAuthChange` drives `renderAuthState`, toggling `#btnSignIn`/`#signedInInfo` and showing the signed-in display name/email; `completeEmailLinkSignInIfPresent()` runs once at boot to finish an in-progress email-link sign-in. `#btnStats` opens `#statsOverlay` (`openStatsOverlay()` fetches via `fetchUserPlays()` and renders through `statsView.ts`'s helpers), independent of sign-in state — see key decisions below. `#btnCalendar` opens `#calendarOverlay` (`openCalendarOverlay()`); `startDailyGame(targetDifficulty, date, dailyButton)` (generalized from a `todayUtc()`-only signature for #41) is shared by `#btnDaily`, `#btnDailyRandom`, and a calendar-day click alike.

## Key design decisions

- **Notes as bitmask** — `notes[r][c]` is a 9-bit integer; bit `i` set means digit `i+1` is pencilled in. Toggle with XOR, clear with AND-NOT.
- **Hint cells become given** — `applyHint` marks the revealed cell as `given: true` so it becomes uneditable, matching standard Sudoku UI conventions.
- **Unique-solution guarantee** — `countSolutions(puzzle, 2)` is called after each removal during generation; cells are only removed when exactly one solution remains.
- **Undo does not re-increment mistakes** — snapshots capture `mistakes` at the time of the move, so undoing a wrong entry also restores the mistake count.
- **Solver tests cover `isValid` only** — `solve` and `countSolutions` are backtracking algorithms that are too slow to run against constrained boards in unit tests.
- **`getPeerCoords` deduplicates row/col/box** — box loop uses `r !== row && c !== col` to avoid duplicating cells already covered by the row and column sweeps (20 peers, not 24).
- **Yarn 4 standalone project** — an empty `yarn.lock` at the project root is required because `~/package.json` exists with a `packageManager` field; without it Yarn treats the project as a sub-workspace.

## Pause timer (2026-07-11)

- **`started`/`paused` on `GameState`** — the timer doesn't run until the player enters their first digit (`started` flips `true` inside `enterNumber`, only for an actual value, not a pencil note). Every state-mutating action in `game.ts` (`selectCell`, `enterNumber`, `eraseCell`, `undoMove`, `toggleNotesMode`, `applyHint`) no-ops while `paused`, so the pause guarantee holds even if a UI guard is missed.
- **`pauseGame`/`resumeGame`/`togglePause`** — `resumeGame` re-anchors `startTime` to `Date.now() - elapsed * 1000` so the displayed time picks up where it left off rather than including the paused interval.
- **Pause overlay reuses the existing `.overlay`/`.modal` pattern** — same blurred full-screen treatment as the loading/victory overlays, which also has the side effect of blocking board interaction while paused (no extra click-blocking CSS needed). `P` is a keyboard shortcut for pause/resume.

## Playwright e2e tests (2026-07-11)

- **[e2e/sudoku.spec.ts](e2e/sudoku.spec.ts)** — same pilot pattern rolled out from `wordle-helper`: `playwright.config.ts` at the root (`testDir: ./e2e`, `webServer` runs `yarn dev` against port 3110), `test:e2e` script, CI step after `test:coverage` with a cached browser install, `e2e/` excluded from Vitest's glob and covered by `yarn lint`. Not in the pre-commit hook (browser install/startup too slow for a hook), matching the cross-repo rollout plan.
- **Clue counts aren't asserted exactly** — `generatePuzzle` in `generator.ts` stops removing cells early if uniqueness would break, so the actual given-cell count can exceed the `CLUE_COUNTS` target. Tests assert relative counts (expert < easy) instead of the exact 46/36/28/22 figures.
- **Locators re-evaluate on every action** — `.cell:not(.given)` is a live selector; once a hint reveals a cell it gains `.given` and drops out of that locator, silently shifting `.first()`/`.nth()` to a different cell on the next call. Tests that pick a cell and then act on it snapshot `data-row`/`data-col` first, then re-locate by those attributes for follow-up assertions.
- **Wait for `#loading` to be hidden before reading board state** — puzzle generation is async (a 30ms yield plus the actual backtracking work), so a bare `locator.count()` read right after `page.goto()` or a difficulty click can race it and undercount `.cell.given`. Every test that reads board state first asserts `#loading` has the `hidden` class.
- **Pause overlay blocks clicks at the browser level, not just in app state** — Playwright's own actionability check refuses to click the numpad through `#pauseOverlay` (`intercepts pointer events`), which is confirmation the CSS-only blocking approach (documented under Pause timer above) works. The pause test uses `{ force: true }` to bypass that check and additionally confirm `game.ts`'s `paused` guard makes the input a no-op even if the overlay were somehow bypassed.

**Key decisions:**

- **Puzzle ID = content hash of the solved grid, not a sequential counter** — generation is client-side and random (`Math.random()` today), so there's no central sequencer. Hashing the solution means two independently-generated identical puzzles collapse to the same Firestore doc/stats automatically, which is what "we already generated this exact puzzle" requires.
- **Seeded PRNG is the prerequisite for everything else (#15 first)** — both daily puzzles (deterministic seed → same puzzle every time) and dedup/stats (stable ID from a reproducible generation) depend on replacing raw `Math.random()` with a seedable generator. `generatePuzzle(difficulty, seed?)` keeps `seed` optional so existing random-puzzle callers are unaffected.
- **Anonymous auth (`signInAnonymously`) before real login** — split into its own issue (#17) rather than bundled with #12, so play stats have a stable uid to attach to immediately, and can be merged into a real account once #12's login lands, rather than waiting on login to exist first.
- **Client-side doc creation, not a `beforeUserCreated` trigger** — mirrors the `files` repo's Firestore-doc-per-user pattern but avoids the paid Blaze plan requirement, consistent with the storage-backend research already on file (see "Storage backend research done" note above).
- **`recordPuzzleStart`/`recordPuzzleCompletion` never block or fail gameplay** — both are called fire-and-forget from `main.ts` with a `.catch(console.warn)`; a Firestore write failure (offline, rules rejection) should never prevent the player from continuing the puzzle. `completionRecorded` module flag in `main.ts` guards against double-counting if `render()` runs again after `solved` flips true.
- **`ensureAnonymousAuth` is memoized as a single module-level promise** — `signInAnonymously` only fires once per page load; subsequent calls (from `recordPuzzleStart`/`recordPuzzleCompletion`) await the same in-flight/resolved promise rather than re-authenticating.
- **`ensureAnonymousAuth` waits for the first `onAuthStateChanged` emission before deciding to sign in, rather than checking `auth.currentUser` synchronously** — flagged by Copilot on PR #22: `auth.currentUser` is still `null` on a fresh page load even when a persisted anonymous session exists, because restoring it from IndexedDB is itself async. The synchronous check called `signInAnonymously()` on effectively every page load, silently replacing the uid each time and defeating the "stable identity across visits" goal of #17. Also fixed in the same pass: the `onAuthStateChanged` listener is now unsubscribed on every exit path (success, sign-in failure, listener error) rather than only on success, and a rejected `authReady` resets itself to `null` so a later call retries instead of replaying a cached failure for the rest of the page session.
- **`src/firebase.ts` never lets a bad/missing config crash the app** — `getAuth()` throws synchronously on an invalid API key, and since `firebase.ts` is a static import reached from `main.ts` (via `stats.ts`), an uncaught throw there aborted the entire module graph before `init()` ran, leaving a blank page with no board. Caught live via PR #22's CI run: every Playwright e2e test failed identically waiting on `.cell:not(.given)`, because this repo's CI has no `VITE_FIREBASE_*` secrets configured. Fixed by wrapping `initializeApp`/`getAuth`/`getFirestore` in `try`/`catch`; `auth`/`db` are now `| undefined`, `stats.ts` rejects/throws from `ensureAnonymousAuth`/`requireDb` instead, and those rejections are caught and logged in `main.ts` per the "never block gameplay" decision above. Verified locally by running `yarn test:e2e` with `.env.local` removed (matching CI exactly) — all 7 tests pass, with two `console.warn` lines per game instead of a crash. Firebase secrets in CI were considered and rejected as the fix: gameplay shouldn't depend on Firebase being configured at all, in CI or in a fresh local checkout.
- **Display-facing puzzle numbering is explicitly deferred** — the content hash is sufficient for dedup/stats; a separate human-facing sequence number (if wanted) can be layered on later without touching the identity scheme.
- **`puzzle` is stored as 9 row-strings, not a 9×9 nested array** — Firestore rejects nested arrays outright (`Nested arrays are not supported`), confirmed by writing a real doc against the `sudoku-craigmcn` project during #16. Each row is a 9-character string (`'0'` for blank); `Board`'s `(number | null)[][]` shape is only ever a client-side/solver concept, never written to Firestore as-is.
- **`firestore.rules` counter updates are bounded, not just monotonic** — `timesPlayed`/`completions` may increase by at most 1 per write, `totalPlayTimeMs` by at most 6h (21600000ms); added after Copilot's PR #21 review flagged that `>=`-only checks let a client jump a counter to an arbitrary value in one write, not just increment it. Also added `is int` checks and a `hasOnly()` key restriction on `puzzles` create (a client could previously attach arbitrary extra fields). `puzzleId`-not-validated-as-real-hash and `dailyPuzzles`-values-not-existence-checked are both documented in the rules file as accepted trust boundaries rather than gaps to close — not practically fixable through Firestore rules without disproportionate cost for a hobby-scale project.
- **Firestore rules aren't deployed via CLI** — no Firebase CLI wiring in this repo; `firestore.rules` is the source of truth and gets pasted into the Firebase console manually on every change. Confirmed working end-to-end via a local scratch script (not committed) that signs in anonymously and exercises both valid and rules-violating writes against the real project.
- **Daily puzzles are generated locally by every client, never fetched from Firestore to play** — `generateDailyPuzzle(date, difficulty)` calls the same `generatePuzzle(difficulty, seed)` used everywhere else, seeded from `dailySeed(date, difficulty)` (a single-pass FNV-1a hash of `"${date}:${difficulty}"`). Since generation is a pure function of the seed, every player's browser independently computes the byte-identical puzzle — `dailyPuzzles/{date}` in Firestore is bookkeeping (for a future admin/listing feature), not a dependency of gameplay. No `firestore.rules` changes were needed for #18 — the `dailyPuzzles`/`puzzles` shapes it writes were already covered by #16's rules (including the `dailyDate` key on `puzzles` create).
- **"Daily random" reuses one of the day's easy/normal/hard puzzleIds rather than generating a fifth puzzle** — `dailyRandomDifficulty(date)` deterministically picks one of `DAILY_RANDOM_POOL = ['easy', 'normal', 'hard']` (expert excluded, matching the issue's "approachable shared challenge" intent) via the same seed-hash technique; `cacheDailyPuzzles` reuses that difficulty's already-computed puzzleId as `random` instead of a separate generation.
- **`cacheDailyPuzzles` tolerates a `permission-denied` race on the `dailyPuzzles/{date}` create** — `firestore.rules` makes that doc create-once (`allow update: if false`), so if two clients both request an uncached date at once, the loser's `setDoc` fails with `permission-denied`. That's treated as success rather than surfaced as an error, since the value it would have written is identical by construction (deterministic generation) to whatever the winner wrote.
- **`cacheDailyPuzzles` checks `dailyPuzzles/{date}` existence *before* generating** — flagged by Copilot on PR #23: the `getDoc` check originally ran after all 4 puzzle generations + `ensurePuzzleDoc` transactions, so every daily-button click paid that full cost even once the date was already cached (this function fires on every click, not just the first). Reordered so the existence check short-circuits the whole loop in the common case; `dailyPuzzle.test.ts`'s "already exists" test now asserts `ensurePuzzleDoc` wasn't called either, not just `setDoc`.
- **`startNewGame()` in `main.ts` has a sequence-token guard against overlapping calls** — also flagged by Copilot on PR #23: it yields 30ms before generating (so loading UI can render), and with no guard an older, slower-to-resolve call could overwrite a newer one's `state` with a stale puzzle. `#btnDaily`/`#btnDailyRandom` made this more likely to matter by adding two more entry points that mutate the same shared `difficulty`/`activeSeed` state before calling it. Fixed with a module-level `gameGeneration` counter — each call captures its own value and bails after the yield if a newer call has since started. Verified manually with a Playwright script that fires two overlapping calls (Hard + Today's Puzzle in quick succession) and confirms the last click wins; no unit test added since `main.ts` has no existing test file (established repo convention).
- **`ensurePuzzleDoc` extracted from `stats.ts` into `src/puzzleDoc.ts`** — `recordPuzzleStart` and `cacheDailyPuzzles` both need "create `puzzles/{puzzleId}` if it doesn't exist yet"; pulling that transaction logic (plus `boardToRows`/`requireDb`) into a shared module avoided a second copy that would drift from the first.
- **`main.ts`'s `activeSeed` module variable threads a daily seed through `createGame`** — `#btnDaily` seeds today's puzzle for the currently-selected difficulty; `#btnDailyRandom` additionally switches `difficulty` to `dailyRandomDifficulty(date)` first. Any regular difficulty button, New Game, or Play Again resets `activeSeed` to `undefined` (and clears both daily buttons' `.active` state), matching pre-#18 expectations that those actions always give a fresh random puzzle rather than silently staying pinned to today's daily seed.
- **`hashSolution`'s two passes now use different multipliers, not just different offset bases** (#20) — `fnv1a(str, offsetBasis, prime)` takes the prime as a parameter; pass one keeps the standard FNV-1a 32-bit prime (`0x01000193`), pass two uses MurmurHash2's multiplier (`0x5bd1e995`) instead of reusing FNV-1a's. Sharing a multiplier across both passes meant they were the same function reseeded, not two independent hashes — weaker combined collision resistance than the ~64-bit fingerprint implied, which matters once a collision would silently merge two distinct puzzles' Firestore stats (#16 made this load-bearing). **This changes every previously-computed `hashSolution` output** — any `puzzles/{puzzleId}` docs already written against the real `sudoku-craigmcn` project (from earlier live-testing during #16/#17/#18, not real player data) are now orphaned under the old IDs; acceptable since the project has no real users yet.
- **`generator.test.ts` determinism is now also tested on `'expert'`** (#20) — the seed threads through both the fill step and the `countSolutions`-driven removal step, and `'expert'` removes the most cells, exercising the removal path hardest; the original test only covered `'easy'`, which removes the fewest.

## Persist in-progress game state (2026-07-20)

Implements issue #29 — a backgrounded mobile (and sometimes desktop) tab that the browser discards and reloads from scratch had no saved state to rehydrate from, so it always looked identical to starting over.

**Completed:**

- [src/persistedGame.ts](src/persistedGame.ts) (`saveGame`, `loadGame`, `clearSavedGame`) + tests in `persistedGame.test.ts`; wired into [src/main.ts](src/main.ts) — see the Architecture entries above for both.

**Key decisions:**

- **One `localStorage` slot, not one per puzzle** — only one game is ever in progress at a time in this app, so there's nothing to key by; a single `sudoku-saved-game` entry is simpler than a `puzzleId`-keyed scheme and is fully overwritten by the next `startNewGame()`'s first `render()` regardless.
- **Restored games always come back paused (if already started)** — rather than trusting the saved `paused` flag, `tryRestoreGame()` forces `paused: true` whenever `state.started` is true, so a reload never drops the player straight into a silently-still-ticking timer. A game that hadn't started yet (no digit entered) restores as-is, since `pauseGame` is a no-op pre-start anyway and there's no timer to protect.
- **`render()` is the primary save point, with `visibilitychange`/`pagehide` as a backstop** — nearly every state mutation in `main.ts` already calls `render()`, so hooking `saveGame` there covers the common case for free. The one gap is the timer's per-second `setInterval` tick, which updates `state.elapsed` directly without calling `render()` (see `startTimer`) — the `visibilitychange`/`pagehide` listeners exist specifically to flush that otherwise up-to-a-second-stale `elapsed` value at the moments a tab is most likely to actually get killed.
- **Versioned and age-limited, not just existence-checked** — `SCHEMA_VERSION` guards against rehydrating a save shaped by an older `GameState`; a 24h `MAX_AGE_MS` guards against resuming a puzzle from days ago that the player has likely forgotten about. Both cases, plus a corrupt/unparseable entry, are treated identically: discard the save and clear the key, falling back to `startNewGame()`.
- **Never blocks gameplay on storage failure** — all `localStorage` calls are wrapped in `try`/`catch` (private browsing, quota exceeded, etc.), matching the existing `firebase.ts`/`stats.ts` philosophy that a persistence-layer failure should degrade gracefully, not crash or block play.
- **Verified manually, not just via unit tests** — a scratch Playwright script (not committed) loaded the dev server, entered a digit, confirmed the `localStorage` entry, reloaded the page, and confirmed both the digit and the pause overlay were restored correctly.
- **`saveGame` takes only `state`, not a separate `difficulty` param** — flagged by Copilot on PR #30: the original signature took the module-level `difficulty` from `main.ts` as a second argument, but that variable is updated synchronously on a difficulty-button click while `state` doesn't catch up until `startNewGame()`'s async 30ms yield resolves. A save triggered in that window (e.g. the tab backgrounded mid-click) would tag the *previous* puzzle's state with the *new* difficulty. Since `GameState` already carries its own `difficulty` field, the fix removes the redundant parameter entirely rather than trying to keep two copies in sync — `saveGame`/`loadGame` now read/return it as part of `state`.
- **`saveGame` strips `history` (the undo stack) before writing** — also prompted by the same PR's review: `history` grows by one full board+notes clone on every move and was previously serialized to `localStorage` in full on every `render()`, an unbounded and unnecessary write-size growth over a session. The undo stack isn't needed to satisfy the resume feature (only board/timer/pause state is), so it's dropped from the persisted payload; a resumed game simply starts with an empty undo history.

**Outstanding / next:** none for this issue. Next up per the project board: #26 (per-user play history write path).

## Per-user play history (2026-07-21)

Implements issue #26 — the write path for a per-user play history, building on #17's anonymous identity and #25's real login. UI to read/display this data is #27, deliberately out of scope here.

**Completed:**

- `recordUserPlay(puzzleId, difficulty, mistakes, elapsedMs)` added to [src/stats.ts](src/stats.ts); called from `handleVictory()` in [src/main.ts](src/main.ts) alongside the existing `recordPuzzleCompletion` call, under the same `completionRecorded` once-per-solve guard.
- `firestore.rules`: new `users/{uid}/plays/{puzzleId}` match — read/write restricted to `request.auth.uid == uid`, with the same bounded/typed-field discipline as the existing `puzzles` rules (`hasOnly` on keys, `difficulty in [...]`, integer/range checks on `mistakes`/`elapsedMs`).
- Tests in `stats.test.ts` covering the write path, the doc path (`users/{uid}/plays/{puzzleId}`), the overwrite behavior, and the `db`-not-configured rejection.

**Key decisions:**

- **Doc ID is `puzzleId`, so a repeat play overwrites rather than accumulates** — the issue explicitly asked to decide and document this. Given the collection is already `users/{uid}/plays/{puzzleId}` (puzzleId as the doc ID, not an auto-ID), the natural behavior is "this user's most recent attempt at this puzzle," not a full attempt log. `recordUserPlay` uses a plain `setDoc` (no `{ merge: true }`), so each write fully replaces the previous entry for that puzzle. A full attempt history (auto-ID per play) was considered and rejected as unnecessary for the stats view #27 is expected to build (recent games / per-difficulty counts / best time), all of which only need the latest or best attempt per puzzle, not every attempt.
- **`recordUserPlay` is a separate function from `recordPuzzleCompletion`, called alongside it, not merged into it** — per the issue's own suggestion. They write to different collections (`puzzles/{puzzleId}` aggregate counters vs. `users/{uid}/plays/{puzzleId}` per-user doc) with different semantics (`increment()` vs. full overwrite), so keeping them as two calls avoids one function doing two unrelated kinds of writes.
- **`completedAt` is constrained to `request.time` in rules, not just checked as a timestamp type** — `serverTimestamp()`'s value resolves to `request.time` when the rules evaluate the write, so requiring exact equality prevents a client from submitting an arbitrary/backdated timestamp while still allowing the standard `serverTimestamp()` sentinel through.
- **`allow create, update` share one validation block, not separate ones** — because doc IDs are reused (see the overwrite decision above), a client can't know in advance whether a given write is a Firestore `create` or `update`, so both need to accept the exact same shape. This mirrors the `puzzles` collection's `create` rule's field discipline rather than the counter-bounded `update` rule, since this is a full-document overwrite, not an increment.
- **Manual Firebase Console step required, same as #16/#21/#25**: the updated `firestore.rules` was pasted into Firebase Console → Firestore Database → Rules → Publish for `sudoku-craigmcn`, then verified live via a local scratch script (not committed) signed in anonymously against the real project: a valid write to `users/{uid}/plays/{puzzleId}` succeeds, a repeat write to the same doc overwrites it, and writes are rejected for an extra/unexpected field, an out-of-range `elapsedMs`, an invalid `difficulty`, and — critically — a write attempted against a *different* uid's history path. All six cases behaved as the rules intend.

## Plays rules hardening (2026-07-21)

Closes #32, #33, #34 — three follow-ups from #26/PR #31's review, all touching the same `users/{uid}/plays/{puzzleId}` rule in [firestore.rules](firestore.rules), bundled into one PR since they're one reviewable change to one rule rather than unrelated fixes.

**Completed:**

- **#32** — added an upper bound on `mistakes` (`<= 9999`), matching the `is int && >= 0` checks already in place for the lower bound.
- **#33** — raised the `elapsedMs` ceiling from 6h (21600000ms, copied from the unrelated `puzzles/{puzzleId}` counter-increment bound) to 24h (86400000ms), since this field is a single session's absolute elapsed time, not an increment — an uninterrupted solve session can legitimately run long, and the original 6h bound was silently dropping those sessions' history writes entirely (caught by `console.warn`, never surfaced to the player).

Mobile keeps the full, untruncated name/email text (`.signed-in-label`'s old `max-width: 8rem` was removed entirely, not just widened) and the full-text "Sign out" button — `renderAuthState()` in `main.ts` sets `signedInInfo`'s `aria-label` to the same name/email either way, so screen readers get the identity regardless of which visual (avatar or text) is showing.

**Key decisions:**

- **One set of buttons, not two** — the drawer couldn't be built by duplicating `#btnStats`/`#btnSignIn`/daily buttons with separate mobile-only IDs (duplicate IDs break `getElementById`-based wiring and are invalid HTML). Instead `#secondaryNav` and its children are the single source of truth; only their *container's* CSS positioning changes per breakpoint (normal flow on desktop, `position: fixed` off-canvas on mobile) — the `position: fixed` escape from flex/grid layout is what makes this work without JS reparenting.
- **Stats/sign-in moved out of `.header-right` into `.account-nav`, on both desktop and mobile** — a consequence of the above: since the drawer needs `#secondaryNav` to be one physical container holding *everything* secondary, stats/sign-in couldn't stay in the header on desktop while also living in the drawer on mobile. Accepted as a minor, reasonable desktop reshuffle (grouping "extras" together) rather than pursuing a fragile pixel-matched two-container illusion.
- **`.daily-btn` styling reused for `#btnStats`/`#btnSignIn`/`#btnSignOut`** instead of keeping `.pause-btn` (icon-only) — once sitting in a row/list next to labeled daily-puzzle buttons, matching that labeled-button style reads more consistently than mixing icon-only and icon+label buttons in the same nav.
- **`.hidden` utility class (generalized in #37, just before this issue) is what makes `#drawerBackdrop` work with zero extra CSS** — the backdrop starts `hidden` in markup and gets `.hidden` toggled by `openDrawer()`/`closeDrawer()`, the same pattern every overlay already uses.
- **Escape closes the drawer via its own top-level `keydown` listener, not `handleKeydown`** — `handleKeydown` (arrow-key navigation, digit entry, etc.) returns early when `!state || state.solved`, which shouldn't gate whether the menu can be dismissed.
- **`#signedInAvatarIcon`'s display toggles need the `#id.class` selector, not just `.signed-in-avatar-icon`** — discovered live: the Font Awesome kit's own stylesheet sets `display` on `.fa-sharp` at the same (0,1,0) specificity, and since the kit's stylesheet is injected asynchronously, it can win the cascade over our rule purely by load order. This bit twice: once for the responsive show/hide (desktop shows the icon, mobile hides it) and again for `main.ts`'s photo-vs-fallback-icon toggle (`.hidden` alone silently failed to hide the `<i>` element, the only place in this codebase `.hidden` is applied directly to an FA icon rather than a wrapping button/div). Both fixed with `#signedInAvatarIcon.signed-in-avatar-icon`/`#signedInAvatarIcon.hidden` selectors, which out-specify `.fa-sharp` regardless of load order.

## Tablet/mid-range height (2026-07-21)

Follow-up to #35: at "desktop" width but short viewport height (e.g. a 1024×768 landscape iPad, or a shortish laptop browser window), the fixed 3.5rem cell size made the full header+nav+board+controls stack taller than the viewport, so the numpad (and even "New Game") required scrolling to reach — confirmed live via Playwright by measuring the numpad's `getBoundingClientRect()` against `window.innerHeight` across several viewport sizes before changing anything: 600×800, 1024×768, and 1280×800 all failed.

**Fix:** the existing mobile media query (`@media (max-width: 32.5rem)`) gained a second, OR'd condition — `@media (max-width: 32.5rem), (max-height: 50rem)` — so short-but-wide viewports get the same hamburger/drawer treatment as narrow phones, not just a smaller distinction. `--cell-size` inside that block changed from a pure `100vw`-based formula to `min(calc((100vw - 2.5rem) / 9), calc((100vh - 23rem) / 9))`, taking whichever of width or height is more constraining — needed because the old vw-only formula would compute an oversized cell on a wide-but-short viewport (the exact case this fix targets) rather than shrinking it. The `23rem` constant is an empirically-measured budget for all non-board chrome (header, difficulty-nav, action row, numpad, New Game button, gaps/padding) once `#secondaryNav` has collapsed into the drawer.

**Verified live**: re-ran the same before/after measurement across 600×800, 768×1024, 820×1180, 1024×768, 1280×800, 1440×900, plus extra-short 1440×700 and 900×600 — numpad and "New Game" both fully in-viewport at every size. Confirmed the breakpoint boundary lands exactly at 800px height (900×801 stays desktop, 900×800 switches to the drawer). Full `yarn test:run` (130/130) and `yarn test:e2e` (7/7, default Playwright viewport of 1280×720 — itself under the new 800px threshold, exercising this exact path) re-run afterward, both green.

**Outstanding / next:** none for this issue.

## PR #40 review follow-ups (2026-07-21)

Copilot's review of PR #40 (mobile navbar) flagged four issues, all confirmed valid (verified live, not just taken at face value) and fixed in the same PR:

- **Sign out didn't close the drawer** — every other drawer action (`btnSignIn`/`btnStats`/`btnDaily`/`btnDailyRandom`) called `closeDrawer()` first; `btnSignOut`'s handler didn't. Fixed by adding the same call.
- **`#drawerBackdrop` could get stuck visible after resizing out of drawer mode** — confirmed live: `.drawer-backdrop`'s `position: fixed` rule sits *outside* the responsive media query (it's always-on, visibility controlled only by JS's `.hidden` toggle), so opening the drawer on mobile then resizing/rotating to desktop width left a full-screen dark overlay with no visible hamburger left to dismiss it. Fixed with a `window.addEventListener('resize', () => closeDrawer())` — deliberately `closeDrawer()`, not `dismissDrawer()` (see below), since a resize shouldn't steal focus to `#btnMenu`.
- **Closed drawer's buttons stayed in the tab order** — `transform: translateX` moves an element visually but doesn't remove it from focus/tab order. Fixed by toggling `secondaryNav.inert` in `closeDrawer()`/`openDrawer()` (`src/main.ts`) — but only inert while actually in drawer/mobile mode (`drawerMediaQuery.matches`, mirroring the CSS breakpoint), since on desktop `#secondaryNav` is a normal inline row that must stay interactive regardless of its meaningless-there open/closed state.
- **Signed-in identity's only accessible name was `aria-label` on a plain, non-focusable `<div>`** — fragile across assistive tech. Fixed the common case (real photo) by setting the avatar `<img>`'s `alt` to the user's name/email (a real element's `alt` is a more reliable accessible-name source than a generic div's `aria-label`); the icon-fallback case (no `photoURL`) keeps the container `aria-label` as a documented, less-reliable fallback rather than a full re-architecture.

**Key decisions:**

- **`dismissDrawer()` (focus restore to `#btnMenu`) is separate from `closeDrawer()` (no focus side effect)** — the two behave differently on purpose. `closeDrawer()` alone is called from the five action buttons (which each move focus/context elsewhere themselves — opening an overlay, starting a game) and from the resize listener (which shouldn't touch focus at all). `dismissDrawer()` — `closeDrawer()` plus `btnMenu.focus()` — is used only where the user is explicitly closing the menu without taking another action: `#btnCloseMenu`, the backdrop click, and Escape (itself guarded on `secondaryNav.classList.contains('open')` so pressing Escape elsewhere in the app doesn't unexpectedly yank focus to the hamburger button).
- **No full manual Tab-cycle focus trap** — `inert` on the closed state plus focus move-in (`openDrawer()` focuses `#btnCloseMenu`) and move-out (`dismissDrawer()`) covers the concrete issues raised without hand-rolling Tab/Shift-Tab interception. A conscious, narrower scope than a complete modal-dialog focus trap; flagged to the user before implementing and not objected to.
- **`role="dialog"`/`aria-modal="true"` are set only in `openDrawer()`, not statically in the HTML** — `#secondaryNav` is a plain nav row on desktop, not a dialog; the attributes would be semantically wrong there. Since `openDrawer()` is only ever invoked via `#btnMenu`, itself only visible/clickable in drawer mode, this is safe without an explicit mode check.
- **`drawerMediaQuery` (a module-level `window.matchMedia(...)`) duplicates the CSS breakpoint condition as a string literal** — accepted as a necessary duplication (documented with a comment pointing at `public/styles.css`) rather than trying to read the breakpoint out of CSS at runtime, which would be considerably more complex for one boolean.
- **Verified live**, not just by reading the diff: scratch Playwright scripts (not committed) confirmed sign-out closes the drawer; resizing an open drawer past the breakpoint hides the backdrop and resets `inert`; the drawer is `inert` while closed on mobile but never `inert` on desktop; opening moves focus to `#btnCloseMenu` and sets `role`/`aria-modal`; Escape closes it and restores focus to `#btnMenu`. Full `yarn test:run` (130/130) and `yarn test:e2e` (7/7) re-run afterward, both green.

**Outstanding / next:** none for this issue. All four Copilot findings reacted 👍, replied to with the verdict, and resolved on the PR.

## Calendar view for past daily puzzles (2026-07-23)

Implements issue #41 — a calendar overlay letting a player browse and play a *past* day's daily puzzle (previously `#btnDaily`/`#btnDailyRandom` only ever played *today's*), with a per-day completion indicator.

**Completed:**

- [src/calendarView.ts](src/calendarView.ts) (new, pure/DOM-free) + `MIN_CALENDAR_DATE`/`dailyPuzzleId` added to [src/dailyPuzzle.ts](src/dailyPuzzle.ts) — see Architecture above for both.
- `startDailyGame` in [src/main.ts](src/main.ts) generalized to take a `date` parameter (previously hardcoded `todayUtc()`), shared by `#btnDaily`, `#btnDailyRandom`, and calendar-day clicks.
- `#btnCalendar`/`#calendarOverlay` added to `.daily-nav`/[index.html](index.html), mirroring `#btnStats`/`#statsOverlay`'s structure; calendar-specific CSS in [public/styles.css](public/styles.css) including a mobile-drawer-breakpoint adjustment.
- Tests in `calendarView.test.ts` (pure logic, no mocking, mirrors `statsView.test.ts`) and additions to `dailyPuzzle.test.ts` for `dailyPuzzleId`.

**Key decisions:**

- **Two-state completion status (`completed`/`not-completed`), not the issue's original three-state proposal** — `users/{uid}/plays/{puzzleId}` docs are written only in `handleVictory()` on `recordUserPlay()`; there is no "started but unfinished" doc for any date, so a third state could never be populated for any day except awkwardly "today" via the single global `localStorage` save slot. Decided with the user before implementation rather than guessing.
- **Client-side cross-reference, zero schema/rules changes** — each visible day's puzzle ids are recomputed locally (`dailyPuzzleId`, the same technique `cacheDailyPuzzles` already uses) and checked against `fetchUserPlays()`'s existing per-user play list. No new Firestore field, no `firestore.rules` change, unlike most prior features in this repo.
- **Bounded to `[MIN_CALENDAR_DATE, todayUtc()]`, both in the UI (disabled prev/next and day buttons) and re-checked defense-in-depth in the click handler** — `MIN_CALENDAR_DATE = '2026-07-17'` is the date daily-puzzle support (#18) itself shipped (`git log --diff-filter=A -- src/dailyPuzzle.ts`), since no daily puzzle "existed" before that even though the seed math is technically computable for any date string.
- **Performance-driven scan design** — `generatePuzzle('expert', seed)` benchmarks at ~130ms in this repo; naively generating all 4 difficulties for every visible day would freeze the UI for seconds. The scan (`computeCompletedDates` in `main.ts`) only checks difficulties the user has *ever* completed at all (`difficultiesWithCompletions`), checks cheapest-to-most-expensive with early exit per day, memoizes `dailyPuzzleId` per `(date, difficulty)` for the session, and yields to the event loop every 5 days scanned — guarded by a `calendarRequestId` counter mirroring `statsRequestId`'s stale-async-load guard.
- **A calendar-day click plays the currently-selected difficulty** (same semantics as `#btnDaily` today), not a forced `dailyRandomDifficulty(date)` — decided with the user, keeping one consistent mental model for "which difficulty am I about to play" across both entry points.
- **Verified live**, not just via unit tests: a scratch Playwright script (not committed) confirmed the boundary disabling (today/future days and pre-`MIN_CALENDAR_DATE` days correctly unclickable, `#btnCalendarPrev` correctly disabled from July 2026), and that completing a daily puzzle via a calendar-day click causes that day to render `calendar-day-completed` on reopen — screenshotted both states against the real `sudoku-craigmcn` Firebase project.
- **Ran a whole-file Prettier pass on `index.html`/`public/styles.css`/`src/main.ts` mid-implementation, then reverted it** — those files predate consistent formatting in this repo (no `format:check` script), and `prettier --write` reformatted every pre-existing line, not just the touched ones (289 unrelated deletions in `index.html` alone). Caught before pushing; reverted to original formatting and reapplied only the actual feature diff by hand, consistent with the standing `[[feedback_scoped_formatting]]` guidance from PRs #30/#31.
